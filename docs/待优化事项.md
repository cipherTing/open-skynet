## 使用边界

- 性能风险章节记录当前代码中存在明确性能放大路径的事项；非 P1 工程风险章节可保留架构边界、测试缺口和类型建模等记录，但不把它们当成核心性能问题。
- “全量查询”本身不是问题；离线审计、管理员主动任务、固定小集合和有明确数据边界的查询可以全量执行。
- 每项统一标为“可能会”：表示触发条件和性能放大路径已经存在，但实际影响仍需结合数据量、索引和运行指标确认。
- 所有推荐方式都只是初步探查建议，不代表最终必须采用的优化方向。正式修改前必须重新测量、确认不变量并单独制定方案。

## 已明确排除

- 不把“未来可能部署多 API 实例”作为优化理由。
- 不把所有全量查询一概判为问题。
- 不把单纯代码组织、兼容路由、自动化测试数量或类型建模列入 P1 性能风险；需要保留的事项统一放在后面的非 P1 工程风险章节。
- 不因为代码使用了分批或缓存就自动判定安全；仍需确认总工作量和触发位置。
- 站内通知功能已决定移除，不再为当前通知实现规划性能优化；后续由外部 IM 方案重新设计，历史能力和恢复边界见[站内通知移除与未来 IM 接入参考](./站内通知移除与未来IM接入参考.md)。
